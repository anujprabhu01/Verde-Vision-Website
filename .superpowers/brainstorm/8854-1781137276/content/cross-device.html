Cross-Device Consistency Strategy

The specific causes of the different-device behavior — and what fixes each one.

Root causes diagnosed

Problem: B/A Slider touch vs mouse bugs

The slider attaches separate mousedown, touchstart, mousemove, touchmove handlers. On some Android browsers, touch events fire inconsistently. On iOS, touchmove with passive: false can cause scroll jank.

Fix: Pointer Events API

Replace all mouse/touch handlers with a single pointerdown / pointermove / pointerup listener. One unified API that works identically across mouse, touch, and stylus on all modern browsers.

Problem: Scroll event jank

The scroll path animation, reveal animations, and progress bar all listen to scroll events and do layout-triggering reads (getBoundingClientRect) on every frame. On low-end phones this causes dropped frames.

Fix: IntersectionObserver

Replace scroll listeners for reveal animations with IntersectionObserver — runs off the main thread, zero layout thrashing. The path animation can use CSS @scroll-timeline or a single rAF loop with cached measurements.

Problem: iOS viewport height (100vh)

The hero section uses 100vh. On iOS Safari the browser chrome (address bar) is included in the viewport height but visually covers content — causing the hero to appear cropped or the CTA buttons to be hidden under the nav.

Fix: 100dvh

Replace 100vh with 100dvh (dynamic viewport height) — accounts for the browser chrome on iOS. Supported in all modern browsers; safe to use with a 100vh fallback for older browsers.

Problem: Missing vendor prefixes

Properties like backdrop-filter (nav blur), -webkit-text-stroke, and certain transforms need vendor prefixes to work in older Safari/Chrome versions. Currently added manually (and inconsistently).

Fix: Autoprefixer

Astro runs PostCSS + Autoprefixer automatically. Every CSS property gets the correct vendor prefixes for your target browsers — no manual work, no missed prefixes.

Problem: Canvas grid on low-end devices

The animated canvas grid runs a continuous requestAnimationFrame loop on page load, consuming GPU cycles on devices that are already under load rendering the hero.

Fix: client:idle + reduced-motion

Load as a client:idle island — only starts after everything else is loaded. Also respect prefers-reduced-motion (already partially handled, but will be made complete).
